# 自然回答风格审计（ChatGPT 式文章流）

**范围**：仅审计与建议，不涉及代码修改。  
**日期**：2026-05-08  
**构建**：`Legal_AI/frontend` 下 `npm run lint`、`npm run build` 均已通过。

---

## 1. 当前回答为什么仍像「法律报告」

### 1.1 Prompt 层：强制五段式与固定标题

`Legal_AI/config/legal_prompts.py` 中 `LEGAL_RAG_ANSWER_SYSTEM_PROMPT` / `LEGAL_RAG_ANSWER_USER_PROMPT_TEMPLATE` 明确要求：

- **必须按顺序输出**小节，且 **「小节标题须与下列完全一致，以便前端解析」**。
- 固定为：`1) 一句话结论` → `2) 你现在最该做` → `3) 风险提示` →（可选）`4) 可执行操作步骤` → `5) 法律依据`。
- **第 3 节标题必须原样为「3) 风险提示」**，不得并入第 1 节。
- **「3) 风险提示」须写 2～4 条**；**「2) 你现在最该做」须为行动清单**（非长段落）。

这会直接约束模型输出为 **可预测的编号报告骨架**，即使用户希望「像 ChatGPT 一样先自然开篇」，模型仍被训练成先对齐标题与条数，模板感难以避免。

### 1.2 前端解析与展示：`normalizeAnswer` + `QwenKbAnswerCard`

`Legal_AI/frontend/src/app/new-feature-chat/page.tsx` 中的分段规则与 `legal_prompts.py` 的标题 **一一对应**（如「一句话结论」「你现在最该做」「风险提示」等）。解析后 `QwenKbAnswerCard` 将内容 **拆成多块 UI** 渲染，而不是「一整篇 Markdown 文章」。

### 1.3 卡片 UI：模块边界强于文章流

`QwenKbAnswerCard.tsx` 当前结构（摘要）：

- 外层：`rounded-2xl border … shadow` 大卡片。
- 顶栏：「问题：…」「模型：…」与正文 **硬分隔**（`border-b`）。
- **结论**：独立色块——`border-l-[3px]` + `bg-[var(--app-primary-soft)]/[0.12]` + 固定 **h2「结论」**，视觉上仍是 **左侧强调条 + 浅底提示区**，不像 ChatGPT 正文首段。
- **你现在最该做 / 风险提示 / 可执行操作步骤**：各为 `<section>` + **`border-t` 分隔** + 固定 **h3 文案**，块与块之间 **等间距 `space-y-6`**，阅读节奏是「三个独立章节」，而非一段连续 prose 里的小标题。
- **行动**：`ActionChecklistBlock` 左侧 **编号或对勾**，偏「任务清单 / 合规 checklist」。
- **风险**：`RiskBulletListBlock` 非编号行用 **`AlertTriangle` 图标**，偏「警告模块」。
- **依据**：`hasMeaningfulBasis` 时仍是可折叠的「引用依据」区块；底部另有 **「知识库来源」** 整块。

综合效果：**信息架构 = 报告目录 + 侧栏强调 + 清单**，与 ChatGPT 常见的 **`article` + `prose` + 首段直给答案** 差异明显。

### 1.4 与「自然文章流」目标的张力

您期望：**开头直接回答 → 后面用小标题、加粗、列表、表格组织；不要每块都像独立模块**。  
现状是：**Prompt 强制分段标题 + 前端硬编码章节标题与分隔 + 结论区 callout 样式**，三层叠加后，即使模型文风偏口语，**版式仍会读出「结构化法律报告」**。

## 2. `QwenKbAnswerCard.tsx` 审计与改进方向（建议，非实施）

### 2.1 当前外层结构

- 根：`div` 白底圆角卡片 + 边框 + 轻阴影。
- 子块顺序：问题/模型 → **结论 callout** → 三个 **顶边框 section**（行动、风险、步骤）→ 可选依据折叠 → `KnowledgeSourcesBlock` → 操作按钮行。

### 2.2 结论区是否仍像提示框

**是。** `border-l-[3px]` + 浅色底 + 大标题「结论」组合，在常见设计系统里属于 **highlight / callout**，不是正文第一段。ChatGPT 式体验通常 **不用左侧色条包裹首段**，而是与普通段落同一 `prose` 流，仅用 **字重或极短引导语**（甚至无引导语）突出首答。

### 2.3 行动建议与风险提示是否仍像独立模块

**是。** 原因包括：

- 每节固定 **h3 标签文案**（与产品话术强绑定）；
- 节与节之间 **`border-t` + `pt-6`** 的「章节翻页」感；
- **Check / AlertTriangle** 强化「控件感」与「合规清单感」。

### 2.4 哪些样式让回答「不像 ChatGPT」

| 要素 | 当前表现 | 与 ChatGPT 式 prose 的差异 |
|------|-----------|---------------------------|
| 结论区 | 左条 + 浅底 + h2「结论」 | ChatGPT 很少把首答放在 alert 框里 |
| 章节分隔 | 多处 `border-t`、`space-y-6` | 更像文档分节，而非连续阅读 |
| 固定节标题 | 「你现在最该做」「风险提示」 | 偏产品化模块名；ChatGPT 多用用户可见的 Markdown 标题（由模型生成） |
| 列表装饰 | 对勾、三角警告 | 偏任务/风险 UI，非中性列表 |
| 顶栏「问题：」 | 与答案强分割 | ChatGPT 对话里问题在气泡里，答案区通常只有回答流 |

### 2.5 如何改成 `article` / `prose` 风格（设计建议）

在不改变引用与表格能力的前提下，可考虑的方向（后续实现时再取舍）：

1. **根容器**：用 `<article className="prose prose-neutral … max-w-none">`（或项目里已有的 typography 封装），减少「大卡片套小卡片」层级。
2. **首答**：结论与普通正文 **同一背景**，去掉或弱化左侧色条；可选 **无「结论」h2**，或改为更小、更轻的「摘要」样式（甚至由模型用 `##` 自命名）。
3. **后续块**：用 **模型输出的 Markdown 标题**（若改为整段 Markdown 渲染）替代固定 h3；若仍结构化传参，可把 h3 改为 **`text-sm font-medium` + 更少上下边距**，去掉 `border-t` 或改为 **仅标题下细线**，避免「每节一页」。
4. **行动列表**：可改为 **普通有序/无序列表**（`ol`/`ul`），或保留 checklist 但 **弱化图标颜色与尺寸**。
5. **风险**：用 **中性圆点列表** 替代 `AlertTriangle`，文案侧由 prompt 改为「需要注意」等更生活化表述；图标可选 **完全不用**。
6. **「问题：」行**：可移入对话列表由上层展示，或缩成 **一行 metadata**，降低与正文的视觉竞争。

### 2.6 `[n]` 引用悬浮能否保留

**能。** `renderTextWithCitations` + `InlineCitationMark` + `CitationPopover` 与外层布局 **解耦**；改为 `article`/`prose` 只影响包裹层与排版类名，**不必移除**引用按钮与悬浮卡片逻辑。

### 2.7 `ActionStepsTable` 能否保留

**能。** 表格块可放在 prose 内用 **`overflow-x-auto`** 包裹（当前步骤区已有类似处理）；`ActionStepsTable` + `renderCell` 传入 `renderTextWithCitations` 的路径可保留，仅需统一父级排版类名即可。

## 3. `legal_prompts.py` 审计与改进方向（建议，非实施）

### 3.1 是否仍强制五段式

**是。** 「五、回答结构」规定 **第 1、2、3、5 节强制**；第 4 节按条件输出。用户模板再次列出 **严格顺序与标题**，与前端解析假设一致。

### 3.2 是否导致模板化

**是。** 固定编号标题、强制清单、风险 **2～4 条**、行动 **清单化**、法律依据固定在第 5 节，都会提高 **句式与篇幅的可预测性**。系统提示虽写「像帮朋友理清问题」，但 **结构约束压过文风描述**。

### 3.3 「你现在最该做」是否限制条数

Prompt 要求 **行动清单**，**未写死「最多 N 条」**；用户模板强调 **不得省略、须为行动清单**，条数由模型与场景决定。前端 `splitActionLines` 仅做展示拆分，**无条数上限**。

### 3.4 「风险提示」是否限制条数

**有：2～4 条**（系统提示第 46 行、用户模板第 113 行）。

### 3.5 是否适合把「风险提示」改为「需要注意」

**从产品语气上适合**更生活化；**需注意与前端、解析器、历史会话的兼容**：

- `page.tsx` 中 section 识别大量使用 **「风险提示」** 字面与 `3) 风险提示`；
- 系统提示现要求 **标题必须原样「3) 风险提示」**。

若仅改文案不改解析，会 **断裂**。若改为「需要注意」，需 **同步**：prompt 固定标题、解析正则、`QwenKbAnswerCard` 的 h3 文案，以及旧数据兼容策略。

### 3.6 是否适合要求「开头直接自然回答」而非报告式编号

**适合作为产品目标**，但与当前「标题完全一致以便前端解析」**冲突**。可选演进方向（概念层）：

1. **保留机器可读结构**（如 JSON 或隐藏分隔符）与 **用户可见 Markdown 文章** 分离——改动面大。
2. **放宽标题格式**：允许 `## 直接回答` 等，但 **解析器与 prompt 同步放宽**——中等改动。
3. **仍输出 Markdown，但第一节不要求写「1) 一句话结论」标题**，改为「第一段必须为直接回答，勿写章节号」——需改解析逻辑，避免把首段误判进其他 bucket。

### 3.7 Prompt 调整原则（摘要建议）

- 减少 **「必须原样标题」** 的刚性，或改为 **「推荐结构」+ 软约束**，并与解析策略一起改。
- 风险：**允许 1～3 条** 或「视必要列点，不必凑满」，降低凑数感。
- 行动：**鼓励短句、每条一行、优先 3～5 条可执行动词短语**，避免长篇分身份堆砌（可分身份仍保留，但限制每条长度）。
- 用语：**「需要注意」替代「风险提示」** 若全局改名则需连带解析与 UI。
- 明确鼓励：**首段 2～4 句直接作答**，其后用 Markdown `##` / 列表 / 表格组织，**避免司法文书编号体**（若仍要 `[n]` 引用，可单独强调引用规则不变）。

## 4. `ProcessTimeline.tsx`：回答完成后是否应默认收起

### 4.1 当前行为

- 主折叠：`const [open, setOpen] = useState(true);` → **默认展开**（第 86 行）。
- 依据分析子区：`analysisExpanded` 默认 **`true`（默认全文展开）**（第 390 行）。

**回答完成后没有**根据「流结束 / answer_done」**自动收起**的逻辑；用户手动收起后，主区 `open` 会保持；依据分析区注释说明 **不因新事件重置**用户选择。

### 4.2 是否会抢占回答视觉焦点

**会，尤其在宽屏或流程事件较多时**：时间线默认展开时，**「检索与依据分析」** 占据纵向空间，且内含检索摘要、时间线行、依据分析正文等，**容易先于或并行于答案卡片吸引注意力**（具体还取决于 `page.tsx` 中组件上下顺序，本文件未改，但默认 `open=true` 强化了该倾向）。

### 4.3 是否适合在回答完成后默认收起、只保留摘要

**从产品目标上适合**：回答完成后用户首要任务是 **读答案**；过程面板改为 **一行摘要 +「展开详情」** 更符合 ChatGPT「主内容优先」习惯。

实现时需注意：

- **流式生成中**仍建议默认展开或至少可见进度，避免用户以为卡住。
- **完成后**再自动收起（或默认 `open=false` 仅在存在最终答案时显示摘要行）需定义清晰触发条件（例如收到 `answer_generation_done` 或最终 answer 事件）。

## 5. 推荐分几步修改（实施路线图，仅建议）

以下顺序兼顾 **解析兼容** 与 **视觉降级成本**：

1. **Prompt 与解析对齐评审**  
   决定是「继续固定五标题 + 只改文风」还是「放松标题 / 改章节名（如需要注意）」。**任何标题变更必须先改 `page.tsx` 的 `normalizeAnswer` 相关规则**，否则内容会进错桶。

2. **`QwenKbAnswerCard` 视觉：article / prose**  
   弱化结论 callout；减少 section 的 `border-t` 与固定 h3 的「报告感」；行动/风险列表装饰中性化。

3. **`ProcessTimeline` 默认态**  
   流式结束后面板默认收起或仅展示摘要；生成中保持可展开。

4. **（可选）长文本单一 Markdown 渲染**  
   若后端或解析层能输出单一 `markdown` 字段，卡片改为统一 `ReactMarkdown` + 引用后处理，**最接近 ChatGPT**；代价是解析、引用、表格、占位符的联合测试面最大。

5. **回归与兼容**  
   旧会话、占位文案、空风险/空行动、`PLACEHOLDER_*` 与 `KnowledgeSourcesBlock` 折叠行为一并验收。

## 6. 结论表

| 问题 | 结论 |
|------|------|
| 为何仍像报告 | Prompt 强制五段 + 固定标题与条数；前端分节 + 结论 callout + 清单/警告图标。 |
| `QwenKbAnswerCard` 推荐 | `article`/`prose`、弱化结论条、减少章节边框感、列表中性化；保留 `renderTextWithCitations` 与 `ActionStepsTable`。 |
| Prompt 推荐 | 放松「标题完全一致」与凑条数；首段直答；风险用语可生活化但须联动解析与 UI。 |
| `ProcessTimeline` 完成后 | **建议**默认收起或仅摘要；当前为 **默认展开**，且无完成态自动收起。 |
| 分步修改 | 见第 5 节；优先 **prompt/解析一致**，再 **卡片版式**，再 **时间线默认态**。 |
