> **审计范围**：阅读 `config/legal_prompts.py`、`new_feature_qwen_kb/service.py`、KB 服务、`data/dev_law_chunks.json`、`QwenKbAnswerCard.tsx`；**未修改业务代码**。  
> **日期**：2026-05-05

## 1. 当前回答目标

根据 **`LEGAL_RAG_ANSWER_SYSTEM_PROMPT`** 与 **`LEGAL_RAG_ANSWER_USER_PROMPT_TEMPLATE`**，模型被要求输出固定四段结构（顺序不可省略）：

| 段落 | 标题形式 | 内容与风格要点（来自 prompt） |
|------|-----------|--------------------------------|
| **结论** | `1) 结论` | 生活化、口语；优先第一句话给核心答案；**控制在 2～3 句话**；关键判断句末 `[n]` |
| **依据** | `2) 依据` | 列表项；**每条最多 1～2 句话**，紧扣问题；可点到法规名/章节；**禁止大段照抄法规正文** |
| **风险点** | `3) 风险点` | **2～4 条**；生活化举例（条款无效、程序违法等）；须能由**当前检索到的有效法条直接支持**，否则只能写「依据不足…」类表述 |
| **建议** | 行首 **`建议：`** 或 **`建议 `** | **具体、可执行**；示例引导身份分层（员工/单位）；禁止只剩「建议人工复核」；若需复核须说明原因与材料 |

另：**公开依据分析**（`LEGAL_PUBLIC_ANALYSIS_*`）在流式链路中先于正式回答生成，结构为「问题焦点 / 主要依据 / 辅助依据 / 依据充分性 / 回答边界」，强调法条与问题的对应关系，而非最终答题。

**检索侧**：**`LEGAL_QUERY_REWRITE_*`** 仅产出检索用 JSON（`search_query`、`core_keywords` 等），**不回答法律问题**，目的为提高百炼/本地 KB 召回。

## 2. 当前回答为什么可能不够实用

以下结合 **prompt 约束**、**服务端拼装** 与 **检索数据** 归纳「偏空、偏法条、偏不生活化」的可能成因。

### 结论是否太短

- **是（设计上偏短）**：系统提示明确要求结论 **「本段控制在 2～3 句话以内」**。用户若期待「像律师初次咨询那样」展开案情与策略，会直接感到信息量不足。
- 同时结论句末凡有关键判断需 `[n]`，在引用压力大时模型可能优先「保守短句」，进一步压缩可读叙事。

### 依据是否仍然偏重

- Prompt 已要求依据「不要大段照抄」「每条 1～2 句话」，**但若检索切片本身偏长或模型习惯性复述条文**，观感仍可能偏「法条摘要」。
- **`service.py` 中传给模型的上下文**：`kb_context` 仅为 **`[n]` + 法规正文一句块**（见下文第 3 节）；而 **`ref_lines` 再次包含完整「法规正文」行**，模型输入中存在 **正文重复与拉长**，可能诱导更长引用式复述。

### 风险点是否具体

- 规则要求 **2～4 条** 且 **每条须能由当前有效法条直接支持**。当知识库只覆盖边缘条文或检索偏题时，模型只能重复「依据不足、需补充材料」类句式，**具体风险场景**难以展开（否则违反「不得凭常识杜撰」）。
- 结果：**风险段容易变成模板化合规声明**，而非用户情境下的「你可能踩的坑」。

### 建议是否可执行

- Prompt 已要求「具体、可执行」、身份分层示例、禁止空洞复核；**但若缺少案情要素（期限、是否书面合同、解除理由等）**，模型既不能编造事实，又不能引入库外法律，**只能落到「补充材料 / 咨询专业人士」**，实用性上限被卡住。
- User 模板第 **9** 条在检索不足时 **强制** 写出固定句式 **「当前知识库检索结果不足，建议补充信息或人工复核。」**，多次问答后用户会感到重复、空泛。

### 是否区分用户身份

- **仅在建议段用示例引导**（「如果你是员工/用人单位……」），**没有结构化字段**要求用户先选择身份；模型只能从自然语言问题里猜，猜不到则分层建议缺失。

### 是否主动提示需要补充哪些事实

- 结论中允许「尚缺哪些关键事实」；公开依据分析里有「回答边界」。**但未强制以清单形式列出「请你补充：①…②…」**，依赖模型自觉，表现不稳定。

### 是否过度依赖法条文本

- **是（机制上鼓励依赖库内文本）**：整套设计禁止训练记忆与外部法常识，**唯一可用材料即检索块**。本地测试库极小时，模型除复述片段外几乎无法「展开实用策略」。

### 是否没有把「下一步怎么做」说清楚

- 建议段本应承担「下一步」；若 KB 不足或模型倾向于合规收尾，**建议易与依据/风险同质化**（仍围绕「依据不足、复核」）。前端虽用绿色区块突出建议，但**内容质量仍由上文约束决定**。

## 3. 当前 prompt 的优点和不足

针对 **`LEGAL_RAG_ANSWER_SYSTEM_PROMPT`**（及 user 模板关键点）：

| 维度 | 是否已在规则中体现 | 说明 |
|------|---------------------|------|
| **生活化** | **是** | 开场即要求「像帮朋友理清问题」、结论口语化；依据勿堆砌。 |
| **身份分层建议** | **部分** | 建议段有示例，但无强制结构（如必须包含劳动者/用人单位两段）。 |
| **行动清单** | **部分** | 要求「可执行下一步」，但未规定编号步骤或时间顺序（如「先…再…」）。 |
| **缺失信息追问** | **弱** | 结论可写「尚缺事实」，user 模板第 9 条在不足时 **固定一句**「检索结果不足…」，易替代「针对本案要补充什么」的追问。 |
| **避免空泛「人工复核」** | **是** | 明确禁止只写复核；若需复核须原因与材料。**但若 KB 空，模型仍常只剩这一条路**。 |
| **避免大段复制法条** | **是** | 依据段字数与「不要照抄」均有约束；需配合 **输入侧 ref_lines 不要太冗长** 更易见效。 |

**`LEGAL_PUBLIC_ANALYSIS_*`**：强调严谨对应与边界，**偏「审计说明」口吻**；与主回答的「生活化」并存时，若前端同时展示，用户可能觉得 **前半段像报告、后半段像答题**，风格略分裂。

**`LEGAL_QUERY_REWRITE_*`**：面向 **百炼检索**，不直接优化「回答实用性」；若 `search_query` 偏离用户真实意图，后续再好的回答 prompt 也救不回召回。

## 4. 当前知识库对回答质量的影响

### `data/dev_law_chunks.json` 规模与内容

- **共 4 条**记录：其中 **1 条 `effective_status` 为「已废止」**，`LocalKBService` 仅在「有效」子集上检索，故 **参与召回的有效示例实为 3 条**。
- 主题覆盖（据文件原文）：**合同效力（民法典示例）**、**书面劳动合同订立（劳动合同法示例）**、**民事权利能力（民法典总则示例）**，外加一条用于过滤测试的废止记录。
- **未包含**与用户示例高度相关的专项场景全文（如 **竞业限制期限**、**违法解除**、**试用期社保**、**盖章效力** 等）——文件中亦无对应关键词切片，检索命中往往依赖「泛命中」或兜底片段，**回答极易变成「仅能部分关联 + 大量依据不足」**。

### 检索不到直接法条时

- Prompt **禁止**用库外知识作确定性结论；检索不足时须声明边界并引导复核。
- **模型只能诚实降级**，实用性主观感受会明显下降——**这不是模型 alone 能修复的**，根因是 **证据材料（KB）与问题的匹配度**。

### 正式百炼知识库是否会改善

- **通常会**：真实索引覆盖更多法规与场景，检索命中相关切片后，依据与风险才有法条可「挂」。
- 仍取决于：**索引质量、切片粒度、时效字段是否可靠**，以及 query rewrite 产出的 `search_query` 是否与百炼索引匹配。

## 5. 当前前端展示是否影响实用性

基于 **`QwenKbAnswerCard.tsx`**：

| 维度 | 表现 | 对实用性感知的影响 |
|------|------|---------------------|
| **结论是否突出** | **较突出**：顶部大卡片 + 「结论」标题 + 较大字号 | 有利于第一眼抓住要点 |
| **风险点是否易见** | **较易见**：独立琥珀色卡片、「风险点」标题 | 有利于扫读 |
| **建议是否像行动清单** | **部分**：绿色区块 + 可按空行拆成多块；**非强制编号清单 UI** | 若正文仍是段落散文，仍不像 checklist |
| **知识库来源是否过重** | **默认折叠**，展开后每条含全文摘要 `line-clamp-2` | 默认不抢占阅读；展开后信息量大 |
| **依据分析 vs 正式回答** | 流式页中 **`ProcessTimeline` 含依据分析**，正式 **`QwenKbAnswerCard` 在流结束后才整体出现** | 用户会先长时间阅读「检索过程 + 依据分析」，**正式四段回答注意力可能被前置分析稀释**（心理感知上的「不够直奔主题」） |

**占位符**：`PLACEHOLDER_BASIS` / `PLACEHOLDER_RISK` / `PLACEHOLDER_SUGGESTION` 在解析失败时使用（见 `page.tsx` 中 `normalizeAnswer`），界面会显得「空」，易被误判为「模型没给实用内容」。

## 6. 推荐优化方向（不改代码，仅建议）

| 优先级 | 方向 |
|--------|------|
| **P0** | **补充本地测试知识库**：增加高频场景（劳动解除、竞业限制、试用期、社保、合同形式与盖章等）**多条有效切片**，否则本地无法客观验证「实用性」prompt。 |
| **P1** | **强化 prompt 中「用户身份分层建议」**：例如强制建议段包含「若你是劳动者 / 若你是用人单位」两个小标题，或先列出需确认的事实再建议。 |
| **P2** | **强化「缺少事实时主动追问」**：结论或单独小节用编号列出待补充事实，减少单一一句「检索不足」。 |
| **P3** | **建议区文案导向「你可以这样做」**：与现有「建议：」兼容，可在提示中要求每条建议以动词短语开头（收集证据、书面催告、申请仲裁期限等）。 |
| **P4** | **依据区继续压缩**：除模型侧外，可考虑 **`ref_lines` 输入不含全文重复**（仅元数据 + 短摘录），减轻复述诱因（属后续代码与 prompt 联动）。 |
| **P5** | **正式环境用真实百炼知识库回归**：用同一批测试问题对比本地与线上召回与回答。 |

## 7. 建议测试问题（10 个）

1. 竞业限制协议最多可以约定几年？超过部分是否无效？  
2. 公司能否以「业绩不达标」为由直接辞退员工而不支付经济补偿？  
3. 建立劳动关系是否必须订立书面劳动合同？未订立有什么后果？  
4. 试用期最长可以约定多久？同一用人单位与同一劳动者能否多次约定试用期？  
5. 用人单位在试用期是否必须为劳动者缴纳社会保险？  
6. 合同仅有法定代表人签字、没有盖章，是否一定无效？  
7. 劳动者提前三十日书面通知解除劳动合同，是否还需要单位批准？  
8. 经济补偿按什么工资标准计算，是否包含奖金和津贴？  
9. 加班工资的计算基数如何确定，未约定时如何处理？  
10. 用人单位拖欠工资，劳动者可以通过哪些途径维权（协商、投诉、仲裁）？

## 8. 当前最优先建议

| 问题 | 建议 |
|------|------|
| **先改 prompt 还是先补知识库数据？** | **本地/弱检索环境下优先补知识库（P0）**：当前 `dev_law_chunks.json` 仅 **4 条且有效仅 3 条**，与劳动场景问题严重不匹配时，**prompt 再强也只能反复「依据不足」**。正式环境则同步验证 **百炼索引**。在此之后再迭代 prompt（身份分层、追问清单）性价比更高。 |
| **是否现在做会话历史？** | **对「单次问答实用性」不是刚需**；若产品目标是连续追问、补充案情，再做会话历史。**否则可排在 KB 与 prompt 迭代之后**。 |
| **是否先做性能优化？** | **与质量正交互**：流式链路若很慢，用户尚未读到有用段落已流失耐心。**建议与质量并行**：性能侧已有专项审计时，以「首屏可读后感受」为阈值；**质量侧仍以 KB + prompt 为主**。 |

## 报告路径与结论摘要

- **报告路径**：`Legal_AI_Zhong/docs/new-rag-answer-quality-audit.md`  
- **当前回答质量最大问题**：在 **本地默认知识库极小且场景不全** 的前提下，**RAG 约束 + 风险须有法条支撑** 叠加，易导致 **结论偏短、依据/风险/建议模板化、实用性主观评分偏低**；**输入侧 `ref_lines` 含全文** 也可能加剧「法条感」。  
- **最建议优先优化的 3 项**  
  1. **扩充 `dev_law_chunks.json`（或切到真实百炼 KB）覆盖高频劳动/合同场景**，使检索与问题对齐。  
  2. **迭代 prompt：强制「待补充事实清单」+ 建议段身份分层/行动化句式**，减少单一「检索不足」复读。  
  3. **评估是否缩短或后置「公开依据分析」对用户的注意力占用**（产品层），让四段正式回答更早成为阅读焦点。

## 附：服务端拼装要点（与质量相关）

以下为 **`QwenKBRagService`** 中与回答内容直接相关的拼装逻辑摘要：

| 项目 | 实现要点 |
|------|-----------|
| **`citations` 过滤与编号** | 仅保留 `effective_status` 显示为 **「有效」** 的条目；再 **`_renumber_citations`** 为 `[1]`…`[n]`。 |
| **`kb_context`（传入回答 prompt）** | **`_build_effective_kb_context`**：每个引用仅 **`[i]` + 法规正文 `text`**，块之间空行分隔；无则 `"未提供"`。 |
| **`ref_lines`（可引用来源清单）** | 每条引用 **`_build_ref_lines_block`**：编号 + 法规名称、类型、时效性、公布/生效日期、章节条文、链接、**完整法规正文**。 |
| **`retrieval_query_info`** | **`_format_retrieval_query_info`**：原始问题、法律意图、核心关键词行、实际检索语句、`required_filters` JSON；改写失败时有说明行。 |
| **最终 user prompt 包含** | 用户问题、`retrieval_query_info`、**知识库检索结果（kb_context）**、**可引用来源清单（ref_lines）**；另含模板中的回答要求 1～10 条。 |

**可见**：模型同时看到 **精简版正文块（kb_context）** 与 **带元数据的长清单（ref_lines）**，后者重复正文；对「少抄条文、多讲人话」的目标，存在 **输入噪声** 风险。
